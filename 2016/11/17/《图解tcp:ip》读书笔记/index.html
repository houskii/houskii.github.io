网络层

  1. IP协议

    1. 基础知识

      • IP地址属于网络层地址
      • 数据链路的地址可以被抽象化为IP地址
      • IP为了简化和提速，采用面向无连接。在发包之前，不需要建立与对端目标之间的连接

    2. IP地址

      IP地址由32位正整数表示，以每8位一组分成4组。

      由网络地址和主机地址组成，现在基本以子网掩码区分

      192.168.1.44/24 前24位为网络地址，后8位为主机地址

      • 网络地址的分类

        • A类

          首位以0开头的地址，1~8位是网络地址。范围在0.0.0.0~127.0.0.0

          一个网络可容纳主机地址上限为2^24-2

        • B类

          前两位为10开头的地址，1~16位是网络地址。范围在128.0.0.0~191.255.0.0

          一个网络可容纳主机地址上限为2^16-2

        • C类

          前三位为110开头的地址，1~24位是网络地址。范围在192.0.0.0~223.255.255.0

          一个网络可容纳主机地址上限为2^8-2 = 254

        • D类

          开头为1110的地址，1~32位是网络地址。范围在224.0.0.0~239.255.255.255

          常用于多播

      • 广播

        在同一个链路中互联的主机之间广播发送数据包，广播地址为IP地址中网络地址部分全部为1

        即在172.20.0.0/16的网络中，广播地址为172.20.255.255

        广播分本地广播和直接广播，不同网络之间的广播叫直接广播

      • IP多播

        多播用于将包发送给特定组内的所有主机

      • 子网掩码

        现在受子网掩码的区分，将子类网络细分出比A类、B类、C类粒度更小的网络。

        它对应IP地址网络地址部分的位全部为“1”，对应主机地址的位全部为“0”

        即172.20.100.0/26对应的子网掩码为255.255.255.192

      • 全局地址与私有地址

        对没有连接互联网的独立网络中的主机，只要保证在本网络中内地址唯一，可以不用考虑互联网即可配置相应的IP地址。即出现了私有网络的IP地址

        • A类私有IP

          10.0.0.0~10.255.255.255 (10.0.0.0/8)

        • B类私有IP

          172.16.0.0 ~ 172.31.255.255 (172.16.0.0/12)

        • C类私有IP

          192.168.0.0~ 192.168.255.255 (192.168.0.0/16)

        在此之外的IP地址称为全局IP。

        如果配有私有IP地址的主机联网时，则通过NAT进行通信

    3. 路由控制

      路由控制表的形成方式有两种：

      1. 管理员手动设置，也叫静态路由控制
      2. 路由器与其他路由器相互交换信息时自动刷新，也叫动态路由控制

      • 默认路由

        默认路由指路由表中任何一个地址都能与之匹配的记录，为0.0.0.0/0

      • 主机路由

        IP地址/32 称为主机路由。意味着基于主机上网卡上配置的IP地址本身，而不是基于该地址的网络地址部分进行路由

      • 环回地址

        使用127.0.0.1作为还回地址，使用这个地址时数据包不会流向网络

    4. IP报文分片

      由于数据链路不同，MTU也不同。IP报文的最大MTU为2^16-1。在目标主机端重组分片IP数据报成为现在的规范，但是对路由器负担加重

      • 路径MTU发现

        路径MTU（Path MTU）指从发送端主机到接收端主机之间不需要分片时最大MTU的大小，即路径中存在的所有数据链路中最小的MTU。

        路径MTU发现从发送主机按照路径MTU的大小将数据包分片后进行发送，可以避免在中途的路由器上进行分片处理

      发送主机与路由器以FDDI连接，MTU = 4352，而以太网MTU=1500

      路由器会丢弃主机第一次发送的大小4352的包，然后通过ICMP的不可达消息将数据链路上MTU的值发给主机（即1500）

      随后主机分片后再以1500的包发给路由器

    5. IPv6

      IPv6地址长为128比特，一般写成8个16位字节，如FEDC:BA98:7642:3215:DCF3:BA98:6723:CB39

      在IPv6中，如果出现连续的0可以将0省略并用两个冒号::隔开，一个IP地址中仅允许出现一次两个连续的冒号

    6. 首部长度

      IPv4首部长度为20字节

  2. ARP

    ARP，地址解析协议（Address Resolution Protocol）

    实际通信时，必须同时知道IP地址和MAC地址（IP地址有层次性而MAC地址没有，仅凭MAC地址无法确定计算机的位置）。ARP借助ARP请求与ARP相应两种类型的包确定MAC地址。

    在网络A中，主机1想知道主机2的MAC地址，要通过广播发送一个ARP请求包，包中包含想要了解MAC地址的IP地址，目标IP与本身IP一致时，主机2将自己的MAC地址塞入ARP响应包返回给A

    为了避免不必要的网络流量，会把获取到的MAC地址缓存一段时间，即把第一次通过ARP获取到的MAC地址作为IP对MAC的映射关系存到一个ARP缓存表中

    • RARP

      Reverse ARP，即通过MAC地址获取IP地址

    • 代理ARP

      通过Proxy ARP将ARP请求转发给临近的网段

  3. ICMP

    网络控制消息协议（Internet Control Message Protocol），用于在IP主机、路由器之间传递网络通不通、主机是否可达、路由是否可用等网络本身的控制信息

    主机A给主机B发送一个ARP请求包，路由1未能发现主机B的存在，于是给主机A发送一个ICMP包，说明发往主机B的包未能成功

    ICMP的通知消息分两类

    1. 通知出错原因的错误信息
    2. 用于诊断的查询信息

    以明文的形式通过IP进行传输

    主要的ICMP消息

    • ICMP目标不可达

    • ICMP重定向消息

      路由器发现发送端主机使用了次优的路径发送数据，它会返回一个ICMP重定向消息给主机，包含最合适的路由信息和源数据。

    • ICMP超时消息

      IP包中有一个字段叫TTL（Time To Live）生存周期，它的值随每经过一个路由器就会减1，直到减到0该IP包就会被丢弃，同时给主机发送一个ICMP超时的消息，通知该包已经被丢弃

    • ICMP回送消息

  4. DHCP

    动态主机配置协议（Dynamic Host Configuration Protocol）

    使用前需要先假设一台DHCP服务器，将所要分配的IP地址、相应的子网掩码、路由控制信息、DNS服务器地址设置到服务器上。

    • 工作原理

      1. DHCP客户端广播发送DHCP发现包，要求设置IP地址和子网掩码；

        DHCP服务器接受该包并向客户端返回DHCP提供包，通知可以使用的网络设置

      2. DHCP客户端广播发送DHCP请求包，通知服务器想要使用的设置；

        DHCP服务器接受该包并向客户端返回DHCP提供包，通知允许客户端的设置

      这样保证即使重复设置也能正常工作

    在大规模组织机构的网络环境中，一般会设置DHCP中继代理

  5. NAT

    1. Network Address Translator 网络地址转换，是用于在本地网络中使用私有地址，在连接互联网时专用全局IP地址的技术。还出现了可以转换TCP/UDP端口号的NA

      • 工作机制

        • NAT

          以10.0.0.10的主机与163.211.120.9的主机进行通信为例，利用NAT，途中的NAT路由器将发送源地址从10.0.0.10转换为全局的IP地址202.244.174.37再发送数据。反之，当包从地址163.211.120.9发过来时，目标地址202.244.174.37先被转换成私有IP地址10.0.0.10再被转发

          在NAT路由器内部，有一张自动生成的用来生成转换地址的表。当10.0.0.10向163.211.120.9发送第一个包时生成这张表，并按照表中的映射关系进行处理

        • NAPT

          目标主机163.211.120.9的端口号是80，LAN中有两个客户端10.0.0.10和10.0.0.11同时进行通信，并且这两个客户端的本地端口都是1025。如果仅转换IP地址为全局地址202.244.174.37，则转换后所有数字完全一致。所以只要将10.0.0.11的转换为202.244.174.37:1026就可以解决。

    • 潜在问题

      • 无法从NAT的外部向内部服务器建立连接
      • 转换表的生成与转换操作都会产生一定开销
      • 一旦NAT遇到异常需要重新启动时，所有TCP连接都将会被重置，即使使用两台NAT做容灾备份

    • 解决

      1. 使用IPv6
      2. 在NAT内侧主机上为了生成NAT转换表，需要先发送一个虚拟的网络包给NAT的外侧。NAT会照样读取包首部中的内容并自动生成一个转换表。如果转换表构造合理，就可以实现NAT外侧主机与内侧主机建立连接。应用还可以通过一定的方法将NAT路由器上附属的全局IP地址传给应用。NAT外侧与内侧可以进行通信，这种现象叫做NAT穿越

  6. RIP

    Routing Information Protocol 是距离向量型的一种路由协议，广泛用于LAN。使用UDP传输

    RIP将路由控制信息定期（30s）向全网广播一次，如果等待了6次仍未收到路由信息，会关闭路由之间的连接。

    RIP基于距离向量算法决定路径。距离（Metrics）的单位为“跳数”，指经过的路由器的个数。根据距离向量生成距离向量表，再抽出较小的路由生成最终的路由控制表

    路由表中最长距离不超过16，防止无限计数问题

  7. OSPF

    Open Source Path First 是一种链路状态型路由协议，使用IP传输。

    包括OSPF在内的SPF路由协议都基于Dijstra算法

    OSPF中，路由器之间交换链路状态生成网络拓扑信息，然后再根据这个拓扑信息生成路由控制表。

    RIP的路由选择，要求途中经过的路由器个数越少越好。与之相比，**OSPF可以给每条链路赋予一个权重，并始终选择一个权重最小的路径作为最终路由。

  8. IP相关技术

    • IP隧道

      为了让属于IPv4和IPv6的网络之间可以正常通信，在IP首部后面继续追加IP首部。这种在网络层首部后面继续追加网络层首部的通信方式叫做“IP隧道”

    • IP任播

      IP任播是指为那些提供同一种服务的服务器配置同一个IP地址，并与最近的服务器进行通信的一种方法。例如DNS根域名服务器。

      IP任播无法保证第一个包和第二个包发送给同一个主机

    • Mobile IP

      • 显式拥塞通知

传输层

  1. 端口号

    1. 定义

      数据链路和IP中的地址，分别指MAC地址与IP地址。MAC地址用来识别同一链路中不同的计算机，后者用来识别TCP/IP网络中互连的主机和路由器。传输层中类似地址的概念，就是端口号

    2. 端口号如何确定

      • 标准既定
      • 时序分配

    3. 端口号与协议

      端口号由其使用的传输层协议决定。不同的传输协议可以使用相同的端口号。数据到达IP层后，会首先检查IP首部的协议号，再传给相应协议的模块。

  2. UDP

    User Datagram Protocol

    UDP利用IP提供面向无连接的通信服务，没有流量控制和重发。

  3. TCP

    Transmission Control Protocol

    是一种面向连接的通信服务，可以进行丢包时的重发控制，还可以对乱序的分包进行顺序控制。

    TCP的特点

    • 连接管理

      TCP通过三次握手（three-way handshake）创建一个连接：

      服务器端打开一个套接字socket然后监听来自另一方的连接，这就是常指的被动打开（passive open）。服务器被被动打开以后，用户端就能创建主动打开

      1. 客户端通过向服务器发送一个SYN来创建一个主动打开，客户端把这段连接的序号设为随机数x

        此时客户端进入SYN-SEND状态

        包：SYN = 1 ，seq = x

        ​

      2. 服务端回送一个SYN/ACK，确认码ack应该为x+1，SYN/ACK包本身又有一个随机序号y

        此时服务端进入SYN-RECEIVE状态，客户端接收到SYN/ACK后进入ESTABLISHED状态

        包：SYN = 1，ACK = 1 ，ack = x+1 ，seq = y

        ​

      3. 最后，客户端再发送一个ACK，服务端收到这个ACK时完成握手。此时包序号被设定为x+1，响应为y+1

        此时服务端进入ESTABLISHED状态

        包：ACK = 1，ack = y +1 ，ses = x+1

      ​

      TCP通过四次挥手来关闭连接：

      由于TCP连接是全双工的，因此每个方向都必须单独进行关闭。这个原则是当一方完成它的数据发送任务后就能发送一个FIN来终止这个方向的连接。收到一个 FIN只意味着这一方向上没有数据流动，一个TCP连接在收到一个FIN后仍能发送数据

      客户端和服务端都处于ESTABLISHED状态，此时发起主动关闭的为客户端

      1. 客户端发送一个FIN给服务端，来关闭它们之间的数据传送，客户端把这段信息的序号设为随机数u。

        此时客户端进入FIN-WAIT-1阶段。

        包：FIN=1，seq=u

        ​

      2. 服务端收到这个FIN，它发回一个ACK，确认码ack为收到的序号+1，即ack=u+1，包本身随机序号seq为v。

        此时服务端进入CLOSE-WAIT状态；客户端收到后进入FIN-WAIT-2状态

        包：ACK=1 ，seq = v ，ack = u+1

        ​

      3. 服务器准备关闭与客户端的连接，发送一个FIN给客户端。确认码ack为收到的序号+1，即ack=u+1，包本身随机序号seq为w。

        此时服务器进入LAST_ACK状态

        包：FIN = 1，ACK =1，seq = w，ack = u+1

        ​

      4. 客户端收到这个FIN，发回报文ACK确认，并将确认序号ack设置为收到的序号+1，即ack=w+1，包本身序号为FIN包的确认码，即seq = u + 1

        此时客户端进入TIME-WAIT状态，在关闭连接之前，客户端至少需要等待2个MSL（Max Segment Lifetime），随后关闭和服务端的连接。服务端收到FIN，关闭与客户端的连接

        包：ACK = 1，seq = u+1 ， ack = w+1

      Q&A

      Q1：为什么断开连接时客户端要有TIME-WAIT状态？

      A：TIME_WAIT是TCP协议用以保证被重新分配的socket不会受到之前残留的延迟重发报文影响的机制,是必要的逻辑保证

      ​

      Q2：断开连接时，服务端还未发送FIN包时，是否可以继续向客户端发送数据？

      A：可以

      ​

      Q3：为什么连接的时候是三次握手，关闭时却是四次挥手？

      A：服务端在确认客户端想要断开连接的请求时，可能存在数据还未传输完毕的情况，先回复ACK报文，在数据传输完成后再发送FIN报文

      ​

      Q4：断开连接时，服务端是否可以将ACK和FIN包一起发送给客户端？

      A：可以

    • 通过序列号与确认应答提高可靠性

      TCP通过肯定的确认应答（ACK）实现可靠的数据传输。当发送端将数据发出之后会等待对端的确认应答。如果有确认应答，说明数据已经成功到达对端。但反之，也非100%是数据丢失，还有可能是数据已经收到但应答丢失。

      而确认应答处理、重发控制、重复控制这些功能都可以通过序列号和确认应答号来实现，TCP借此实现可靠传输。

    • 窗口控制与重发控制

      TCP以1个段为单位，每发一个段进行一次确认应答的处理，但如果每次都等到收到确认才再次发送，包往返时间越长性能越低。

      因此TCP引入了窗口的概念，发送端主机在发送了一个段以后不必要一直等待确认应答，而是继续发送。窗口大小就是指无需等待确认应答而可以继续发送数据的最大值。这个机制实现了使用大量的缓冲区，通过对多个段同时进行确认应答的功能。

      在滑动窗口以外的部分包括尚未发送的数据以及以及确认对端收到的数据。当数据发出口若如期收到确认应答就可以不用再进行重发，此时数据就可以从缓存区清除。收到确认应答的情况下，将窗口滑动到确认应答中的序列号位置。这种机制也成为滑动窗口机制

      使用了窗口控制机制，某些确认应答即便丢失也无需重发。

      ​

      当某一报文段丢失后，发送端会一直收到序号为报文段首序号的确认应答。如数据1001~2000丢失，则确认应答会一直发送序号为1001的确认应答。在窗口比较大又出现报文段丢失的情况下，同一个序号的确认应答将会被重复不断地返回，而发送端主机如果连续3次收到同一个确认应答，就会将其所对应的数据进行重发。这种称为高速重发控制。

    • 流控制

      为了防止在高负荷情况下无法接受任何数据，导致数据被丢弃而触发重发机制。TCP提供流控制机制，让发送端根据接收端的实际接受能力控制发送的数据量。

      具体操作是，接收端主机向发送端主机通知自己可以接收数据的大小，于是发送端会发送不超过这个限度的数据。这个限度就被称为窗口大小。如果窗口的更新通知在传送途中丢失，可能会导致无法继续通信。为了避免此类问题的发生，发送端主机会时不时发送一个叫做窗口探测的数据段来获取

    • 拥塞控制

      如果在通信刚开始或者网络出现拥堵时突然发送一个较大的数据，极有可能会导致整个网络的瘫痪。

      首先为了在发送端调节要发送的数据量，定义了一个叫做“拥塞窗口的概念”，在慢启动时，将这个窗口大小设置为1个数据段（1 MSS）发送数据，之后每收到一次确认应答，拥塞窗口的值就加1。发送数据包时，将拥塞窗口大小和接收端主机通知窗口大小做比较，选择较小的值，发送比其还小的数据量。

      在超时重发时，会设置为当前拥塞窗口一半的大小。

应用层

  • DNS

    IP地址对于用户来说并不方便记忆，因此在使用TCP/IP进行通信时，系统会将一串由罗马字和点号组成的字符串转换为IP地址。这就是DNS（Domain Name System）的功能。

    在应用中，当用户输入主机名时，DNS会自动检索那个注册了主机名和IP名的hosts数据库，并迅速定位对应的ip地址。

    域名服务器：根部所设置的DNS叫做根域名服务器，它分管下一层的域名服务器。域名和域名服务器按照分层来设置

    解析器：进行DNS查询的主机和软件叫做DNS解析器。
