  1. TCP的标志比特有哪几种？

    • URG：紧急指针(urgent pointer)有效
    • SYN：同步序号用来发起一个连接
    • FIN：发送端完成发送人物
    • ACK：确认序号有效
    • RST：重建连接
    • PSH：接收方应该尽快将这个报文段交给应用层

    存在

    四次握手的情况：

    当Client和Server同时主动打开与对方建立连接时。

    由于双方不知道对方会与自己建立连接，则两端会同时向对方发送SYN包，同时进入SYN_SENT状态，在收到对方发送的SYN包后进入SYN_RCVD状态并发送ACK包，由于之前自己已经发送了SYN包，所以现在只需要等待对方对自己SYN的回应ACK，并在收到回应时进入ESTABLISHED状态。

    三次挥手的情况：

    当Server收到Client发送的FIN包时，自己回应ACK包时已经没有需要传输的数据，则Server会将ACK和FIN一次性发送给Client，并直接进入LAST_ACK状态。Client对Server发送的FIN包进行回应进入TIME_WAIT状态，Server收到Client的回应后进入CLOSED状态，断开连接。

  5. 三次握手时客户端没有ACK怎么办？

    当Client发送SYN给Server后断开了连接，Server收到并返回了SYN&ACK，却一直没有收到Client返回的ACK，这时连接既没有建立也不算失败。因此需要一个重试时间来让Server决定是否放弃这个连接，否则就会一直占用Server的SYN连接队列直至耗尽。

    Linux中默认5次重发SYN&ACK包，时间间隔是前一次的两倍，即为1s / 2s / 4s / 8s / 16s，加上第五次发送后等待的32s，即一共63s。

    由于Server需要63s才能断开这个连接，如果在63s内大量只发送SYN队列，也会引发问题（即SYN flood攻击）。linux提供了几个TCP参数：tcp_syncookies、tcp_synack_retries、tcp_max_syn_backlog、tcp_abort_on_overflow 来调整应对。

    ​

  6. 两端同时断开连接时网络是什么状态？

    两端同时向对方发送FIN包后进入FIN_WAIT_1，在收到对方发来的FIN包后，向对方发送ACK包并进入CLOSING状态。由于对方已经发送了FIN包表示断开连接，因此没有数据包流入。在收到对方发送对自己FIN包的ACK包后，进入TIME_WAIT状态，等待2MSL后断开连接

  7. 哪一端会进入TIME_WAIT？TIME_WAIT有什么用？

    发送FIN包的一段会进入TIME_WAIT状态，如果两方都发送了则都进入（如Q5）

    TIME_WAIT是为了防止发送方在给对方的FIN包发送ACK后直接断开连接。如果这个ACK包在途中遗失，则接收方无法收到ACK包，同时因为链接已经断开，接收方要求发送方重新发送ACK包的FIN包请求也无法正确到达。如果这个端口恰好连入一个新的发送方，可能会发生一下两种情况：

    1. 由于未建立连接时收到一个FIN包，违反了传输协议，新发送方会返回一个RST，接收方本应该正确关闭的连接却收到了一个错误。
    2. 刚好建立了连接后收到一个FIN包，又恰好这个FIN包有一定概率符合这个连接的配置，使连接莫名其妙关闭。

    同时对于客户端来说，如果端口（MAX = 65535）大量被占用，TIME_WAIT可以作为一个端口释放的缓冲时间
